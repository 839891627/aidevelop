## 1. 先建立全局认知：你到底在构建什么

这个项目本质上是一个“AI 应用工程能力学习系统”，不是单点技术演示。  
它解决的是：如何把大模型从“能聊天”升级为“能查知识、能调工具、可观测、可优化、可演进”的系统能力。

你可以把整条学习主线理解为 6 个连续阶段：

1. **对话层能力**：让模型可交互（阻塞式 + 流式）
2. **模型层能力**：多供应商接入与切换（解耦模型）
3. **提示词层能力**：Prompt 管理与稳定输出
4. **执行层能力**：Function Calling 调用后端工具
5. **知识层能力**：RAG 检索增强与检索质量优化
6. **治理与演进能力**：成本、日志、评估、Agent Loop

一句话记忆：
**Chat 是入口，RAG 是知识，Function 是行动，Observability 是护栏，Agent Loop 是演进方向。**

## 2. 系统架构蓝图：六层模型

### 2.1 交互层（UI / API）
- 用户界面：聊天、成本看板、表单自动填充
- 接口能力：聊天、检索、评估、成本统计、健康检查

学习重点：
- 对外能力抽象是否清晰
- API 是否能稳定承载前端交互节奏（尤其流式）

### 2.2 编排层（Controller / Service）
- 负责“流程组织”而不是“复杂业务堆积”
- 把请求分发给 Chat、RAG、Function、Cost 等领域服务

学习重点：
- 编排层是否过重
- 是否保持服务职责单一

### 2.3 智能层（LLM Runtime）
- `ChatClient` / `ChatModel` 统一模型调用
- 注入系统提示词、历史上下文、检索结果、工具定义

学习重点：
- “调用模型”只是动作，质量关键在“上下文装配”

### 2.4 知识层（RAG）
- 文档加载、分块、Embedding、向量存储
- 查询预处理、检索融合、重排序、策略选择

学习重点：
- 检索系统不是单 API，而是流水线工程

### 2.5 业务数据层（Domain / Persistence）
- 业务实体与仓储（贷款、还款、风控、日志）
- 提供工具调用所需的数据可信来源

学习重点：
- 工具返回的数据结构是否稳定可消费

### 2.6 治理层（Observability / Optimization）
- 调用日志、成本统计、缓存、定时任务、评估指标
- 提供“优化证据”，避免盲目调参

学习重点：
- 是否能回答“为什么变好了/变差了”

## 3. Chat 对话系统：从能用到好用

## 3.1 Chat 的最小闭环
用户输入 -> 构建上下文 -> 调用模型 -> 返回结果 -> 保存会话

这个闭环看似简单，但每一环都影响最终质量：
- 上下文缺失 -> 回答跑偏
- 历史过长 -> 成本升高、延迟增加
- 历史过短 -> 多轮理解断裂

## 3.2 阻塞式与流式（SSE）的取舍

### 阻塞式（一次性返回）
适合：
- 接口调用
- 需要完整响应元数据（如 token、model）
- 后端批处理场景

不足：
- 用户等待感明显

### 流式（逐片返回）
适合：
- 聊天 UI
- 实时交互体验
- “边生成边消费”场景

不足：
- 统计信息获取和错误恢复更复杂

### 判断建议
如果你在做“人机交互体验”，优先流式；  
如果你在做“系统调用链稳定性”，先阻塞式。

## 3.3 多轮对话的核心：上下文窗口治理

### 必知概念
- Token 是成本与窗口的基础单位
- 窗口是有限资源，历史越多不一定越好

### 实践策略
1. 滑动窗口限制历史条数
2. 保留系统消息，裁剪普通历史消息
3. 长对话场景引入“摘要记忆”而非全量记忆

### 常见坑
- 只按“消息条数”裁剪，不按 Token 预算裁剪
- 不区分关键上下文与噪音上下文

## 4. 多模型接入（Multi-LLM）：解耦与选型

## 4.1 为什么必须支持多模型
- **成本差异**：不同模型价格差一个数量级很常见
- **能力差异**：推理、中文、代码、工具调用表现不同
- **稳定性差异**：可用性、限流策略、延迟波动不同
- **合规差异**：不同场景可能要求不同供应商

## 4.2 技术实现原则
1. 业务层只依赖抽象接口
2. 模型供应商由配置和 Profile 决定
3. 不把 provider 逻辑散落在业务代码里

## 4.3 模型选型思路（实战）

可以按任务类型做路由：
- 日常问答：低成本高吞吐模型
- 复杂推理：高质量推理模型
- 结构化提取：输出稳定性更高的模型
- 大规模 embedding：成本可控且中文效果好的 embedding 模型

### 评估维度建议
- 准确率/事实性
- 工具调用成功率
- 平均延迟
- 单次成本
- 稳定性（错误率）

## 5. Prompt 工程：最便宜的质量杠杆

## 5.1 Prompt 的工程化目标
- 可维护：提示词独立管理
- 可迭代：可热更新、可版本对比
- 可评估：变更有前后效果数据

## 5.2 System Prompt 的核心职责
1. 角色定义（你是谁）
2. 能力边界（能做/不能做）
3. 决策框架（遇到不同问题怎么做）
4. 输出规范（格式、语气、引用要求）

## 5.3 Prompt 设计高频策略
- Role + Constraints（角色 + 约束）
- Few-shot（示例引导格式）
- 结构化输出约束（JSON/列表/固定字段）
- 错误与不确定性表达规则（不知道时怎么答）

## 5.4 实战判断：该先改 Prompt 还是改代码？

优先改 Prompt 的场景：
- 输出风格混乱
- 结构化格式不稳定
- 同类问题回答口径不一致

优先改代码/系统的场景：
- 事实错误（知识源不正确）
- 工具调用错误（参数、路由问题）
- 延迟/成本异常（系统策略问题）

## 6. Function Calling：让模型具备“行动能力”

## 6.1 本质理解
LLM 不直接操作数据库，它做的是“工具决策”。  
真正执行动作的是你定义的函数（工具）。

## 6.2 标准流程
用户意图 -> 模型判断是否需工具 -> 选择工具和参数 -> 后端执行 -> 返回结构化结果 -> 模型组织回答

## 6.3 工具设计规范（关键）
1. **描述清晰**：工具用途边界明确
2. **输入可验证**：参数格式可校验
3. **输出结构化**：便于模型二次组织
4. **错误可感知**：失败时返回可解释错误
5. **权限可控**：敏感操作需二次确认

## 6.4 典型失败模式
- 工具描述含糊，模型选错工具
- 参数语义不清，模型传错字段
- 返回长文本，模型再解析不稳定
- 工具异常直接抛出，导致整条链路失败

## 6.5 解决策略
- 使用强约束参数 schema
- 对参数做后端兜底校验
- 结果返回“结构化对象 + 可读摘要”
- 为工具调用添加重试与降级策略

## 7. RAG：从“能检索”到“检索工程”

## 7.1 RAG 的价值边界
RAG 适合解决“事实型、知识库型”问题，尤其企业私有知识。  
它不是万能答案：推理复杂问题仍依赖模型与策略。

## 7.2 RAG 基础流水线
离线：文档 -> 清洗 -> 分块 -> embedding -> 入库  
在线：查询 -> 检索 -> 注入上下文 -> 生成回答

## 7.3 三个关键参数
- `chunk`：决定知识粒度与噪音水平
- `topK`：决定召回覆盖和上下文长度
- `threshold`：决定召回门槛（召回 vs 精度平衡）

### 快速经验
- `topK` 太小：漏知识
- `topK` 太大：噪音高、成本高
- 阈值太高：召回断崖
- 阈值太低：垃圾片段涌入

## 7.4 进阶优化组件

### Query Rewrite
解决代词、省略、上下文指代问题。  
典型触发：`它`、`这个`、`那种情况`、超短查询。

### Query Expansion
通过同义词和术语扩展增加召回覆盖。  
适合领域术语多、用户口语化输入强的场景。

### BM25
补齐向量检索对精确词的弱项（编号、术语、关键短语）。

### Hybrid Search
融合语义检索与关键词检索，提升综合表现。

### Rerank
先粗召回，再交给模型精排，提高前几条结果质量。

### Pipeline
按查询特征自动选择上述组件组合，减少手工策略分支。

## 7.5 如何决定用哪种策略（实用决策）

可用一个简化判断：
- 查询含编号/专有词 -> 优先 Hybrid
- 多轮对话含代词 -> 先 Rewrite
- 答案准确性要求高 -> 加 Rerank
- 通用问答、低延迟优先 -> Vector Only

## 7.6 评估体系：没有评估就没有优化

建议至少维护以下指标：
- Recall：是否找全
- Precision：是否找准
- F1：综合平衡
- MRR：首条正确结果位置
- NDCG：整体排序质量

评估流程建议：
1. 建立标注样本集（query -> relevant docs）
2. 跑不同策略组合
3. 记录指标 + 延迟 + 成本
4. 基于证据选择默认策略

## 8. 成本、日志、缓存：工程可持续性的核心

## 8.1 为什么必须从早期就做治理
不治理的典型后果：
- 成本不可控
- 故障难定位
- 优化无依据

## 8.2 成本治理框架
- 采集每次调用 token 和模型信息
- 建立按日/周/月统计
- 观察模型分布与峰值异常
- 做阈值告警

## 8.3 可观测性框架
每次调用至少应可追踪：
- trace/session
- 模型与 provider
- 请求与响应耗时
- token 与成本
- 状态（成功/失败/超时）

## 8.4 缓存策略框架
建议按“数据性质”分缓存：
- AI 响应缓存（重复问答）
- 检索结果缓存（重复查询）
- 工具结果缓存（重复参数调用）

缓存治理要点：
- TTL 与业务时效匹配
- 键设计包含关键参数
- 提供显式失效机制

## 9. Agent Loop：如何升级成“可执行系统”

## 9.1 为什么不是直接多 Agent
多 Agent 在早期常见问题：
- 复杂度暴涨
- 可观测性变差
- 调试和评估成本高

所以建议：先单 Agent MVP，再拆分角色。

## 9.2 单 Agent 最小闭环
1. Plan：生成下一步计划
2. Act：调用工具
3. Observe：记录结果
4. Reflect：判断是否继续
5. Respond：输出最终答案

## 9.3 最小状态模型（建议）
至少维护：
- `traceId`
- `stepIndex`
- `toolCalls`
- `observations`
- `finished / reason`
- `finalAnswer`

## 9.4 关键护栏
- `maxSteps` 防无限循环
- 工具白名单与参数校验
- 超时与重试策略
- 失败降级（回到直接回答）

## 9.5 评估 Agent 的 4 个核心指标
- 工具调用正确率
- 首答正确率
- 平均步骤数
- 平均时延/成本

## 10. 学习路线（查阅型版本）

### 第一阶段：打通基础闭环
目标：跑通聊天、流式、多轮历史
产出：你能清楚解释 Chat 的完整调用链路

### 第二阶段：建立可控性
目标：多模型切换 + Prompt 迭代
产出：你能做 A/B 对比并解释差异原因

### 第三阶段：建立执行与知识能力
目标：Function Calling + RAG 基础进阶
产出：你能让系统“查知识 + 调工具”协作完成任务

### 第四阶段：建立工程能力
目标：成本、日志、缓存、评估
产出：你能用数据驱动优化而非凭感觉调参

### 第五阶段：进入 Agent 化
目标：单 Agent Loop MVP
产出：你能跑通 Plan-Act-Observe-Reflect 闭环

## 11. 高频问题与排障清单（实战可查）

## 11.1 回答不准
优先排查顺序：
1. Prompt 是否限制不清
2. RAG 是否召回不足
3. 工具是否没触发或参数错误
4. 模型是否不适配该任务

## 11.2 成本过高
优先排查顺序：
1. 历史窗口是否过大
2. `topK` 是否过高
3. 是否缺缓存
4. 是否过度使用高价模型

## 11.3 延迟过高
优先排查顺序：
1. 是否启用了多重策略（Hybrid + Rerank）
2. 工具调用是否慢查询
3. 是否可以先流式返回降低感知延迟
4. 是否可降级到轻量策略

## 11.4 工具调用不稳定
优先排查顺序：
1. 工具描述是否歧义
2. 参数字段命名是否可理解
3. 是否缺参数校验与兜底
4. 失败是否有重试和回退

## 15. 考试笔记版（速记 + 面试问答）

> 用法建议：先看“30 秒速记”，再看“可背要点”，最后做“高频问答”自测。

### 15.1 全局架构

**30 秒速记**
- 六层：交互、编排、智能、知识、业务数据、治理
- 主线：Chat 入口，RAG 知识，Function 行动，Observability 护栏，Agent 演进

**可背要点**
1. 编排层做流程，不做重业务逻辑堆叠  
2. 智能层质量关键在上下文装配，不在“调用 API”  
3. 治理层决定系统是否可持续优化  
4. RAG 是检索工程，不是单接口  
5. 业务层提供可验证的数据真相来源  
6. 架构目标是“可替换、可追踪、可迭代”

**高频问答**
- Q：为什么 AI 应用需要分层？  
  A：降低耦合，便于替换模型、优化策略、定位问题。
- Q：哪一层最容易被忽略但最影响上线质量？  
  A：治理层（日志、成本、评估、缓存）。

---

### 15.2 Chat 与上下文管理

**30 秒速记**
- 问答质量 = 模型能力 x 上下文质量  
- 流式提升体验，阻塞式提升稳定统计能力

**可背要点**
1. Chat 最小闭环：输入 -> 上下文 -> 推理 -> 输出 -> 存档  
2. 流式适合 UI，阻塞适合系统接口  
3. 多轮管理要做窗口治理（条数 + token）  
4. 长对话优先摘要记忆，不是无脑保留全历史  
5. System Message 必须稳定且可维护  
6. 先查上下文问题，再怀疑模型能力

**高频问答**
- Q：为什么同一模型有时答得好有时差？  
  A：上下文输入不同，尤其历史与提示词质量波动。
- Q：何时优先流式？  
  A：人机交互、体验优先场景。

---

### 15.3 多模型接入（Multi-LLM）

**30 秒速记**
- 业务依赖抽象，不依赖供应商  
- 用 Profile/配置切换，不改业务代码

**可背要点**
1. 多模型核心动机：成本、能力、稳定性、合规  
2. 统一抽象接口是解耦前提  
3. 模型路由应基于任务而非个人偏好  
4. 选型至少看：准确率、延迟、成本、稳定性  
5. 模型切换能力是工程韧性的关键  
6. 避免 provider 逻辑散落业务层

**高频问答**
- Q：怎么避免供应商锁定？  
  A：统一接口 + 配置化路由 + 能力对比评测。
- Q：为什么“能切换”比“接得多”更重要？  
  A：切换能力决定故障恢复和成本优化空间。

---

### 15.4 Prompt 工程

**30 秒速记**
- Prompt 是最低成本调优杠杆  
- 先做外部化，再做实验化

**可背要点**
1. System Prompt 决定角色、边界、输出规范  
2. Prompt 必须版本化和可回滚  
3. Few-shot 常用于提升格式稳定性  
4. 约束输出能显著降低后处理成本  
5. Prompt 改动要配评测样本做 A/B  
6. Prompt 解决“表达问题”，不解决“知识缺失”

**高频问答**
- Q：什么时候不该继续调 Prompt？  
  A：事实错误来自检索/工具/数据源时，应先改系统链路。
- Q：Prompt 最容易犯的错是什么？  
  A：写成长篇愿望清单，缺少可执行约束。

---

### 15.5 Function Calling

**30 秒速记**
- LLM 决策，后端执行  
- 工具必须“可描述、可校验、可降级”

**可背要点**
1. Function Calling 让系统从“回答”变“执行”  
2. 工具描述清晰度直接影响调用正确率  
3. 入参要强约束，出参要结构化  
4. 敏感操作加确认与权限控制  
5. 工具失败必须可降级，不拖垮主链路  
6. 工具调用要纳入日志和评估体系

**高频问答**
- Q：为什么强调结构化输出？  
  A：便于模型稳定消费并减少二次解析失败。
- Q：工具调用不稳定先查什么？  
  A：工具描述、参数 schema、后端校验与错误处理。

---

### 15.6 RAG（基础 + 进阶）

**30 秒速记**
- 基础：建库 + 检索 + 注入 + 生成  
- 进阶：Rewrite、Expansion、Hybrid、Rerank、Pipeline

**可背要点**
1. RAG 用于补齐时效与私有知识  
2. `chunk/topK/threshold` 决定检索质量上限  
3. Vector 擅长语义，BM25 擅长精确词  
4. Hybrid 是通用优选，Rerank 提升前列质量  
5. Pipeline 价值在自动策略选择  
6. 没有评估指标就不能谈优化

**高频问答**
- Q：什么时候优先用 Hybrid？  
  A：有专有名词、编号、术语精确匹配需求时。
- Q：什么时候优先做 Query Rewrite？  
  A：多轮指代、省略表达、短问题语义不全时。
- Q：评估最少看哪几个指标？  
  A：Recall、Precision、MRR、NDCG（加 F1 作为综合）。

---

### 15.7 成本与可观测性

**30 秒速记**
- 不能测量就不能优化  
- 成本治理要和功能开发并行

**可背要点**
1. 每次调用需记录：模型、token、延迟、状态、trace  
2. 成本统计至少有按日趋势和模型分布  
3. 缓存是降低成本和延迟的第一手段  
4. 缓存键设计错误会导致脏数据风险  
5. 预警机制让问题“提前暴露”  
6. 优化必须以指标变化为依据

**高频问答**
- Q：为什么说日志是 AI 应用的生命线？  
  A：没有日志就无法定位质量、成本、稳定性问题。
- Q：缓存最常见错误是什么？  
  A：键设计不完整或失效策略缺失，导致错答。

---

### 15.8 Agent Loop

**30 秒速记**
- 单 Agent MVP：Plan -> Act -> Observe -> Reflect -> Respond  
- 先闭环，再复杂化

**可背要点**
1. Agent 的关键是“决策 + 执行 + 反思”闭环  
2. 必须设 `maxSteps` 防循环失控  
3. 每轮应有 `traceId` 便于回放  
4. 工具白名单 + 参数校验是安全底线  
5. 先做单 Agent，再谈多 Agent 协同  
6. Agent 优化必须走评估集回归

**高频问答**
- Q：为什么不建议一开始做多 Agent？  
  A：复杂度和调试成本过高，难以建立稳定基线。
- Q：Agent 成功的最小衡量标准是什么？  
  A：工具调用正确率、首答正确率、平均步骤数、平均时延/成本。

---

### 15.9 高频面试总问（跨章节）

1. Chat、RAG、Function 的边界是什么？如何协同？  
2. 你如何设计一个“可切换模型而不改业务代码”的架构？  
3. 当回答不准时，你的排查顺序是什么？  
4. 如何平衡 RAG 的召回率、精确率、延迟、成本？  
5. 为什么 Agent 系统必须先做观测再做优化？  
6. 如何设计 Agent 的防失控机制？  
7. 缓存在 AI 系统里既是优化手段也是风险点，为什么？

---

### 15.10 考前冲刺（5 分钟版）

**必须记住的 12 句话**
1. 大模型应用先做系统，不是先堆模型。  
2. 上下文质量决定回答上限。  
3. Prompt 先外部化，再 A/B。  
4. Function Calling 是执行能力，不是生成能力。  
5. RAG 是检索工程，不是向量接口。  
6. Vector 和 BM25 是互补关系。  
7. Rerank 是“先广后精”的质量提升器。  
8. 指标不全，优化全靠感觉。  
9. 成本治理必须前置。  
10. 没有 traceId，就没有可回放排障。  
11. Agent 先做单闭环 MVP。  
12. 能持续迭代的系统才是好系统。

### 16.7 RAG 一眼图（进阶）

```text
Query
  ↓
Rewrite(可选)
  ↓
Expansion(可选)
  ↓
Vector + BM25 (Hybrid)
  ↓
Rerank(可选)
  ↓

```

策略判断：

```text
代词/省略多 -> Rewrite
术语/编号多 -> Hybrid(BM25+Vector)
准确率要求高 -> Rerank
延迟敏感 -> 轻量策略(Vector only)
```

口诀：**召回看 Hybrid，排序看 Rerank。**
