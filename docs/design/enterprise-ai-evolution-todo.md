# 企业级 AI 系统架构演进 TODO（Embedding 升级暂缓）

## 1. 背景与目标

当前项目已经具备对话、RAG、Prompt Registry、Chat Memory 持久化等核心能力，具备学习与演示价值。  
但如果目标是企业生产级可用（稳定、可观测、可治理、可审计、可扩展），仍需系统化演进。

本规划明确两点：
- **短期策略**：Embedding 模型升级先暂缓，不做模型替换和大规模迁移。
- **推进方式**：先补齐生产基础设施，再进入 Embedding 平滑升级阶段，降低整体风险。

---

## 2. 演进原则

- **稳定优先**：先治理可用性和风险，再做效果优化。
- **小步快跑**：每个阶段都能独立上线、可回滚。
- **证据驱动**：所有优化均有指标基线和回归验证。
- **兼容迁移**：避免一次性重建索引或切断旧能力。

---

## 3. Embedding 升级暂缓期间的必做项（P0）

> 目标：在不更换 embedding 模型的前提下，先把检索系统做成“可运营”的工程系统。

### TODO-EMB-01：建立检索评测基线
- 建立最小评测集（建议 50~100 条业务问答样本）
- 指标：Recall@K、MRR、NDCG、命中率、平均响应时延
- 输出每日/每周趋势，作为后续升级对照基线

### TODO-EMB-02：统一分块策略与版本标识
- 固化 chunk 策略（chunk size、overlap、分隔符规则）
- 增加 `chunk_version` 与 `embedding_version` 元数据字段
- 确保“同一知识库可追溯到哪套切分/向量策略”

### TODO-EMB-03：完善索引生命周期管理
- 增量入库、重建、回滚流程标准化
- 大文档变更时支持按文档粒度重建，避免全量重算
- 增加索引健康检查（文档数、向量数、孤儿分片检测）

### TODO-EMB-04：补齐检索观测与告警
- 记录查询耗时拆解：重写/检索/重排/生成
- 增加“空召回率、低相似度占比、超时率”告警
- 对热点 query 建立慢查询分析

## 4. 企业级 AI 系统升级清单（全局）

以下为除 Embedding 之外，建议同步推进的企业级能力升级。

### 4.1 可靠性与弹性（P0）
- 对外部模型调用增加超时、重试、熔断、限流
- SSE 流式通道增加断连恢复和幂等防重
- 关键链路（Chat/RAG/Prompt）提供降级路径

### 4.2 安全与合规（P0）
- API 鉴权、RBAC、租户隔离（如适用）
- Prompt/消息/日志中的敏感信息脱敏
- 完整审计日志：谁在何时发布了哪个 Prompt 版本

### 4.3 可观测性与成本治理（P0）
- 统一 traceId 贯穿 Controller -> Service -> Provider
- 维度化成本看板：模型、接口、租户、会话
- 预算阈值和自动告警（防成本失控）

### 4.4 数据治理与生命周期（P1）
- Chat Memory 与 Prompt Registry 增加归档/TTL 策略
- 大表分区与冷热分层（按时间/租户）
- 备份恢复演练（RPO/RTO 目标）

### 4.5 评测与发布治理（P1）
- 建立离线回归集：聊天质量、RAG 命中、工具调用准确率
- Prompt 发布引入准入门禁：评测通过后方可发布
- 灰度发布 + 一键回滚流程

### 4.6 多环境与工程化（P1）
- dev/staging/prod 全链路配置隔离
- 关键 SQL 与配置变更纳入迁移脚本（Flyway/Liquibase）
- CI 增加集成测试和契约测试

### 4.7 高阶能力（P2）
- 多向量库或混合检索引擎抽象（便于后续替换）
- Agent Loop 执行轨迹回放与质量评估闭环
- 统一工具调用注册中心（权限、配额、审计）

## 5. 分阶段落地计划（建议）

## Phase A（1~2 周）：生产地基加固
- 完成 P0 项：可靠性、安全、观测、Embedding 基线
- 交付物：告警面板、评测脚本、链路日志、运行手册
- 验收标准：
  - 关键接口错误率可观测
  - 有可执行的故障降级策略
  - 有可复用的检索质量基线

## Phase B（2~3 周）：治理与发布体系
- 完成 P1 项：数据生命周期、发布门禁、灰度回滚
- 交付物：评测准入流程、发布 SOP、备份恢复演练记录
- 验收标准：
  - Prompt 发布有质量门禁
  - 数据归档策略可执行并验证通过
  - staging 与 prod 配置严格隔离

## Phase C（后续迭代）：Embedding 平滑升级
- 在已有基线之上评估新 embedding 模型（A/B）
- 双写或双索引比对，达到门槛后切流
- 验收标准：
  - 关键检索指标提升（如 Recall@10、NDCG）
  - 成本与时延在可接受范围
  - 支持无损回滚至旧索引

---

## 6. Embedding 升级触发条件（何时再启动）

满足以下条件后再启动模型升级：
- 检索评测体系稳定运行至少 2 周
- 线上告警与观测数据完整
- 索引重建与回滚流程已经演练
- 当前模型在关键业务问题上已出现明确瓶颈（有数据证据）

---

## 7. 风险清单与缓解

- **风险：** 指标体系不完善导致“升级是否有效”无法判断  
  **缓解：** 先做基线评测与固定测试集，再做模型替换

- **风险：** 一次性改动过大影响线上稳定性  
  **缓解：** 分阶段发布，所有步骤可回滚

- **风险：** 成本上涨不可控  
  **缓解：** 升级前定义成本上限与自动熔断策略
